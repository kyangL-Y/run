## 1. 需求

### 背景
浏览器插件在商家价格页执行“读取房型”时，当前页面已经进入 `https://ebooking.hwht.com/price/manage`，但返回结果仍为 `item_count = 0`。排查链路后确认，问题位于后端商家房型采集流程，而不是前端展示、房型映射或提交改价流程。

### 目标
- 修复商家价格页“读取房型”过早解析导致返回 0 条房型的问题。
- 保持现有基础功能板块不变，包括前端工作流、房型映射逻辑、提交改价链路和其他竞对采集功能。
- 将改动范围收敛在商家房型读取后端服务与对应测试。

### 约束条件
```yaml
时间约束: 本次仅处理当前商家房型读取缺陷，不附带结构调整
性能约束: 允许增加有限等待与轮询，但不能引入长时间阻塞或明显拖慢正常读取
兼容性约束: 兼容 prefer_cdp、cdp_current_page、storage_state 三种现有采集模式
业务约束: 不修改基础功能板块，不改前端交互，不改映射和提交流程
```

### 验收标准
- [ ] 商家价格页已登录且房型表已渲染时，“读取房型”能返回非 0 的房型列表。
- [ ] 页面仅到达价格 URL 但房型表尚未渲染时，采集流程会继续等待可识别房型内容，而不是直接返回空列表。
- [ ] 不修改 `browser-extension/content.js` 的既有读取入口与基础工作流。
- [ ] 相关单元测试通过，覆盖“等待后拿到房型”和“不过早把空价格页当作可采集页面”。

## 2. 方案

### 技术方案
在 `backend/app/services/fliggy_merchant_service.py` 内增加“商家价格页内容就绪”等待逻辑，并把“仅 URL 命中即视为可采集”的判定收紧到“URL 命中且存在房型/价格相关内容”。

实现上分两层：
- 第一层：在 `_extract_price_items_from_page` 调用前增加显式轮询，等待 `room_rows`、价格文本或可解析 body 文本出现。
- 第二层：调整 `_evaluate_authenticated_page` 的 URL 兜底判定，避免页面尚未渲染完成时被误判为 ready。

本次不修改插件前端 payload，不改映射汇总逻辑，不改提交改价链路。

### 影响范围
```yaml
涉及模块:
  - merchant_pricing: 收紧商家价格页就绪判定，避免过早返回空房型
  - tests: 补充商家房型读取等待与就绪判定回归用例
预计变更文件: 2
```

### 风险评估
| 风险 | 等级 | 应对 |
|------|------|------|
| 等待条件过严导致真实可用页面被误判未就绪 | 中 | 采用“房型行 / 可解析价格文本 / body 文本命中”多信号联合判断 |
| 等待时间增加影响读取速度 | 低 | 采用短轮询和上限超时，仅在内容未就绪时追加等待 |
| 修复波及基础功能板块 | 低 | 改动限制在 `fliggy_merchant_service.py` 和对应测试，不动前端与提交链路 |

## 4. 核心场景

> 执行完成后同步到对应模块文档

### 场景: 商家价格页延迟渲染
**模块**: merchant_pricing
**条件**: 用户已登录商家后台，价格页 URL 正确，但房型表是异步渲染
**行为**: 后端采集流程在 URL 命中后继续等待可识别房型内容，再执行房型解析
**结果**: 返回真实房型列表，不再因为过早解析而显示 0 条

### 场景: 基础工作流保持不变
**模块**: browser_extension / merchant_pricing
**条件**: 用户仍通过现有“读取房型”按钮触发读取
**行为**: 前端入口、映射汇总、提交改价逻辑保持原样，仅后端读取稳定性增强
**结果**: 基础功能板块不受影响

## 5. 技术决策

> 本方案涉及的技术决策，归档后成为决策的唯一完整记录

### merchant-pricing-readiness-fix#D001: 以后端就绪等待修复商家房型读取
**日期**: 2026-04-18
**状态**: ✅采纳
**背景**: 当前问题发生在既有“读取房型”链路内部，用户明确要求不要动基础功能板块，因此需要选择最小修复面。
**选项分析**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 后端补充页面就绪等待与判定收紧 | 改动小，保持前端和基础功能稳定，便于快速回归 | 需要细化等待信号，避免误判 |
| B: 改插件为当前页 DOM 直抓或重做读取入口 | 可绕开部分 Playwright 时序问题 | 会改动前端基础工作流，超出本次边界 |
**决策**: 选择方案 A
**理由**: 当前故障集中在后端采集时序，最小修复即可解决问题，同时满足“不动基础功能板块”的约束。
**影响**: 影响 `backend/app/services/fliggy_merchant_service.py` 的采集时序与 `backend/tests/test_fliggy_merchant_service.py` 的回归测试

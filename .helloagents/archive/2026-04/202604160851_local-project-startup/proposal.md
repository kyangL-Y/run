## 1. 需求

### 背景
当前项目为本地 Flask 后端配合浏览器插件的 AI PoC。用户要求直接启动项目，并确认本地服务可访问的健康检查地址与插件服务地址。

### 目标
在不修改业务代码的前提下，使用现有启动脚本拉起后端服务，确认 `127.0.0.1:8000` 正常监听，并验证关键健康接口可访问。

### 约束条件
```yaml
时间约束: 当前会话内完成
性能约束: 仅启动本地必要进程，不额外增加常驻服务
兼容性约束: 使用项目现有 PowerShell 启动脚本和 backend/.venv
业务约束: 不变更业务逻辑；如依赖缺失，仅允许安装 requirements.txt 中已有依赖
```

### 验收标准
- [x] 确认项目的唯一启动入口与监听端口
- [x] 启动脚本执行成功，服务监听 `http://127.0.0.1:8000`
- [x] `http://127.0.0.1:8000/health` 返回可用状态
- [x] `http://127.0.0.1:8000/plugin/service-status` 返回可用状态或明确错误信息

## 2. 方案

### 技术方案
复用根目录现有 `start_project.ps1` 完成启动。该脚本会先检查 `8000` 端口占用，再确保 `backend/.venv` 和后端依赖可用，随后运行 `backend/run_flask_server.py`。启动后使用健康检查接口与插件状态接口做验收。

### 影响范围
```yaml
涉及模块:
  - start_project.ps1: 负责本地服务启动、依赖检查和健康检查等待
  - status_project.ps1: 负责启动前后状态确认
  - backend/run_flask_server.py: Flask 服务与调度线程实际入口
预计变更文件: 0
```

### 风险评估
| 风险 | 等级 | 应对 |
|------|------|------|
| `8000` 端口被其他进程占用 | 中 | 先执行状态检查，若冲突则停止并回报占用 PID |
| `backend/.venv` 缺失依赖 | 中 | 允许脚本按既有 `requirements.txt` 自动安装 |
| 旧 `.flask.pid` 残留导致误判 | 低 | 以端口监听和健康检查结果为准，不以 pid 文件单独判定 |

## 5. 技术决策

### local-project-startup#D001: 复用项目既有启动脚本
**日期**: 2026-04-16
**状态**: ✅采纳
**背景**: 项目已经提供成套 `start/status/stop` 脚本，重复手工拼接启动命令会绕过现有依赖检查与 PID 管理逻辑。
**选项分析**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 执行 `start_project.ps1` | 复用既有检查、日志与 PID 管理，和项目现状一致 | 依赖 PowerShell 环境可用 |
| B: 手工运行 `backend/run_flask_server.py` | 命令更直接 | 容易绕过端口检查、依赖安装和 pid 文件维护 |
**决策**: 选择方案 A
**理由**: 现有脚本已经覆盖端口、依赖、健康检查与日志路径，最符合“保守修改、尊重现有代码”的要求。
**影响**: 仅影响本次运行流程，不变更任何业务模块逻辑

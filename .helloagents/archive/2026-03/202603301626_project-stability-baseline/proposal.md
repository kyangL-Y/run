## 1. 需求

### 背景

当前项目的 Flask 主入口仍可导入，但测试基线已缺失，且 `README.md` 对外宣称了大量实际不存在的模块、脚本、接口与页面，导致排查和使用过程持续出现“文档式假故障”。

### 目标

- 恢复项目的最小测试执行基线
- 为当前真实存在的 Flask 应用补一组最小冒烟测试
- 收敛 README，只保留当前仓库真实实现的能力与启动方式
- 同步方案包与变更记录，形成可继续迭代的稳定基线

### 约束条件

- 不补做 README 中历史遗失的大功能，只修当前高优先级稳定性问题
- 以现有 Flask 路由和现有源码为准，不引入新的架构层
- 修改范围尽量收敛在依赖、测试和文档

### 验收标准

- 项目依赖中显式包含 `pytest`
- 使用项目 `.venv` 可以成功执行新增测试
- 新增测试至少覆盖 Flask 应用导入与关键页面/API 路由存在性
- README 删除或改写所有已确认不存在的能力、脚本与页面说明

## 2. 方案

### 技术方案

- 在 `backend/requirements.txt` 中补入 `pytest`
- 新增一个轻量测试文件，验证 `app.main` 可导入、首页/登录页/关键业务路由可注册
- 收敛 `backend/README.md` 的“快速启动、核心页面、核心接口、脚本说明”，使其与当前仓库真实实现一致
- 更新知识库变更日志，记录本次稳定性基线修复

### 影响范围

- 依赖：`backend/requirements.txt`
- 测试：`backend/tests/`
- 文档：`backend/README.md`
- 知识库：`.helloagents/CHANGELOG.md`

### 风险评估

- 风险 1：README 改写后，部分历史能力说明会被移除
  - 缓解：以当前代码为唯一事实来源，只保留真实可用入口
- 风险 2：Flask 应用导入测试可能受环境变量约束影响
  - 缓解：测试仅验证应用创建与路由注册，不直接访问真实数据库
- 风险 3：`.venv` 未安装新依赖前，测试仍无法运行
  - 缓解：将其作为本次交付的一部分明确执行并验证

## 3. 关键决策

- `project-stability-baseline#D001`：优先恢复测试基线与文档真实性，而不是补回 README 中已漂移的大量历史能力
- `project-stability-baseline#D002`：用最小冒烟测试锁定“应用可导入 + 核心路由存在”，避免在当前阶段引入高成本集成测试

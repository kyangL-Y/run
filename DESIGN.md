# AI PoC 设计文档

## 设计目标

本项目用于承载酒店 OTA 运营相关的 PoC 能力，当前仓库可见主体是浏览器扩展入口。设计目标是把适合浏览器侧使用的运营能力组织为本地可加载扩展，并通过既有后端服务完成鉴权、店铺上下文、竞对配置、价格采集与商家改价相关代理。

根目录只承担项目聚合与知识库入口职责，具体业务实现放在 `apps/extension/` 等子模块内。根目录文档用于说明项目边界、模块关系和交付约束，避免子模块文档与项目级认知脱节。

## 方案选择

当前采用轻量 monorepo 结构：

- `apps/extension/` 保存 Chrome/Edge Manifest V3 扩展源码。
- `.helloagents/` 保存项目知识库、模块记录和变更记录。
- `private-secrets/` 保存本地敏感材料，并由 `.gitignore` 排除。

没有在根目录引入额外构建系统或统一包管理配置，因为当前可见代码以原生扩展静态文件为主，直接加载扩展目录即可验证。后续如果重新接入后端或前端包管理，应优先在对应子模块内声明依赖和测试命令，再决定是否提升到根目录统一编排。

## 关键决策

- 以代码为真实来源，知识库文档只记录已存在的模块职责和约束。
- 根目录保持聚合职责，不把扩展业务逻辑上移到项目根。
- 敏感数据目录、环境文件、浏览器调试产物和本地 CLI 日志必须被 Git 忽略。
- 浏览器扩展的消息桥接能力属于安全边界，任何能携带登录态或 Bearer Token 的通道都必须有来源校验和受控动作白名单。

## 已知限制

- 根目录 `README.md` 仍是简短标题，项目说明主要集中在 `apps/extension/README.md`。
- 当前仓库可见部分缺少统一测试入口，扩展脚本只能先用 JS 语法检查和模块扫描做基础验证。
- `.helloagents/CHANGELOG.md` 中存在历史路径命名不一致记录，例如 `apps/frontend/extension` 与当前 `apps/extension` 并存，需要后续单独清理。
- 部分后端路径只在知识库中被引用，当前工作区未完整暴露对应源码，不能据此推断后端运行状态。
